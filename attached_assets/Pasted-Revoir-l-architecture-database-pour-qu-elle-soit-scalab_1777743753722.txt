Revoir l’architecture database pour qu’elle soit scalable, multi-années et indépendante de Replit.

Objectif :
L’application TRS / Planning DPI doit fonctionner sans limite d’année : 2026, 2027, 2028, etc.
Ne jamais créer une table par mois, par semaine ou par année.
Toutes les données doivent être filtrables par date, semaine, mois, trimestre, année, équipement, produit, lot et activité.

Architecture cible :
- Frontend : React / Next.js déployable sur Vercel
- Backend : API Node.js / Express ou API routes Next.js
- Database principale : Supabase PostgreSQL
- Auth : Supabase Auth ou système compatible JWT
- Storage fichiers : Supabase Storage
- Déploiement beta : Replit possible uniquement pour développement
- Déploiement production : Vercel + Supabase
- Phase future : possibilité de migrer vers serveur entreprise PostgreSQL sans refonte majeure

Exigence majeure :
L’application ne doit pas dépendre de Replit pour fonctionner.
Replit est seulement l’environnement de développement.
Toutes les variables sensibles doivent être dans .env :
- DATABASE_URL
- SUPABASE_URL
- SUPABASE_ANON_KEY
- SUPABASE_SERVICE_ROLE_KEY
- JWT_SECRET
- STORAGE_BUCKET

Créer une architecture facilement externalisable :
- Pas de données stockées localement dans Replit
- Pas de fichier JSON local comme base de données
- Pas de SQLite pour la production
- Toutes les données métier dans Supabase PostgreSQL
- Tous les fichiers planning importés dans Supabase Storage
- Export complet possible en CSV / Excel / JSON
- Backup SQL possible
- Migration possible vers PostgreSQL entreprise

Règles database :
- Utiliser UUID comme clé primaire.
- Utiliser created_at et updated_at partout.
- Utiliser des index sur :
  - date
  - year
  - month
  - week_number
  - equipment_id
  - product_id
  - lot_number
  - status
- Utiliser des foreign keys.
- Utiliser soft delete si nécessaire, jamais suppression définitive des données critiques.
- Prévoir Row Level Security Supabase.
- Prévoir audit trail automatique sur les tables critiques.
- Prévoir migration SQL versionnée.

Je dois pouvoir :
1. Développer sur Replit
